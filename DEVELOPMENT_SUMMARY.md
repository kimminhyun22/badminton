# 콕매치 개발 요약

이 문서는 콕매치와 유사한 동호회/경기 운영 프로그램을 빠르게 만들기 위한 재사용 요약입니다.

## 현재 프로젝트

- 이름: 콕매치
- 용도: 배드민턴 동호회 민턴LIVE 운영, 출석/상태 입력, 자동 대기표 보충, 코트 진행, 승패 입력, 실시간 공유
- 배포: GitHub Pages
- 운영 URL: https://kimminhyun22.github.io/badminton/
- Firebase 프로젝트: `kokmatch-23b31`
- 주요 파일:
  - `index.html`: 메인 앱, 민턴LIVE 운영, 대기표 생성, 코트 진행, 저장/복원
  - `view.html`: 카톡 공유용 실시간 중계 화면
  - `manifest.json`, `sw.js`: PWA 설치와 캐시
  - `icon-192.png`, `icon-512.png`, `apple-touch-icon.png`: 앱 아이콘

## 버전 규칙

코드를 수정하면 `index.html`의 아래 3곳을 함께 올린다.

- 상단 주석의 `APP VERSION`
- `<meta name="app-version" content="...">`
- `const APP_VERSION = '...'`

예: `1.10.83` -> `1.10.84`

## 핵심 운영 철학

1. 현장 운영자는 비개발자라고 가정한다.
2. 버튼 한 번으로 바로 쓸 수 있어야 한다.
3. 좋은 대진을 만들기 위해 사용자가 여러 번 생성 버튼을 누르게 하지 않는다.
4. 품질점수는 단순 점수보다 실전 해석이 중요하다.
5. 불가피한 반복과 실제로 피할 수 있는 문제를 구분한다.
6. 스마트폰 가독성을 최우선으로 둔다.
7. 경기 중에는 복잡한 기능보다 실수 방지가 더 중요하다.
8. 현재 사용자-facing 제품은 민턴LIVE에 집중한다.
   - 핵심 흐름: 클럽 명부 → 출석/상태 → 대기표 보충 → 코트 진행 → 경기 종료 → 다음 경기 배정 → 실시간 공유
   - 민턴LIVE는 유동 명부, 레슨/휴식/귀가, 게스트, 묶음 요청, 예약 대진을 빠르게 반영해야 한다.
   - 팀전 UI는 현재 화면에서 제거하고, 필요한 엔진 코드는 안전한 내부 보류 코드로만 남긴다.
   - 민턴LIVE에서 중요한 정보는 운영 상황판, 진행중 코트, 임박대기, 예비대기, 자유 대기자다.
   - 문서와 메뉴도 민턴LIVE 기준으로 정리해 운영자가 헷갈리지 않게 한다.

## 대진 생성 주요 기준

- 실력 균형: 양 팀 실력 합 차이를 최소화한다.
- 출전 공정성: 인당 목표 게임 수 달성을 가장 중요하게 본다.
- 파트너 다양성: 같은 파트너 반복을 줄인다. 단, 사용자가 지정한 P 파트너는 예외로 본다.
- 재대결 다양성: 같은 4명이 다시 만나는 경기와 완전히 같은 팀 대결을 줄인다.
- 휴식 간격: 직전 라운드 연속 출전을 줄인다.
- 일정 효율성: 불필요한 보완게임을 줄인다.
- 설정 준수: 남복/여복/혼복/보정 경기 조건이 맞는지 확인한다.

## 민턴LIVE 특수 규칙

- `대기중`은 지금 경기할 준비가 됐다는 뜻이다.
- `미응답`은 참석 의사 전 단계이며, `참석예정`은 올 예정이지만 아직 자동 후보가 아니다.
- `레슨중`, `휴식중`, `종료`, `경기중`은 자동 후보에서 제외한다.
- `곧 귀가`, `마지막게임`은 후보에 포함하되 우선순위를 높인다.
- `마지막게임` 선수는 경기 완료 후 자동으로 종료 처리한다.
- 임박대기는 코트 수만큼 유지하고 예비대기는 1경기만 유지한다.
- 예비대기를 과하게 확정하지 않아 레슨 복귀자, 방금 끝난 선수, 새로 온 사람을 반영한다.
- 참석투표 마감 후에는 `참석자만 남기기`로 미응답 회원을 제외해 운영 명단을 가볍게 만든다.
- 회원 게임 신청은 즉시 대진에 넣지 않고 관리자 승인 후 예약 대진으로 반영한다.

## 실시간 중계 화면 설계

카톡 링크로 들어오는 선수들이 주로 궁금해하는 것:

- 내가 참석/대기 상태로 잘 들어갔는지
- 다음 경기 대진이 무엇인지
- 어느 코트로 준비해야 하는지
- MVP 또는 개인 순위

따라서 `view.html`은 아래 순서가 좋다.

1. 다음 준비 경기와 코트 정보
2. 진행 중 코트
3. 출석/상태 확인
4. MVP/핵심 요약
5. 완료 경기와 개인 순위

스마트폰 가독성을 위해 카드형 UI, 큰 선수 이름, 명확한 팀 색상, 접이식 상세 정보가 적합하다.

## 유사 프로그램을 빠르게 만들 때 재사용할 구조

### 1. 데이터 모델

필수 엔티티:

- 참가자: 이름, 성별, 레벨, 상태, 게스트 여부, 묶음 요청, 최근 대기시간
- 경기: 라운드, 코트, 종목, 팀1 선수 2명, 팀2 선수 2명, 승패
- 설정: 코트 수, 자동배정 여부, 마감 모드, 마감 예약
- 기록: 출전 수, 승패, 파트너 카운트, 상대 카운트

### 2. 생성 알고리즘

추천 흐름:

1. 참가자와 설정 검증
2. 여러 후보 대진 생성
3. 각 후보를 품질점수로 평가
4. 가장 좋은 후보 자동 선택
5. 미달 선수 보완
6. 품질 대시보드와 실전 특이사항 표시

### 3. 품질점수 설계

점수는 운영자에게 쉬워야 한다.

- 숫자만 보여주지 말고 이유를 함께 보여준다.
- “불가피한 문제”와 “재배정하면 개선 가능한 문제”를 나눈다.
- 실전 특이사항에는 사람 이름과 경기 수를 직접 보여준다.

### 4. UX 원칙

- PC: 입력과 결과를 한 화면에서 빠르게 확인
- 모바일: 하단 탭, 큰 버튼, 큰 선수 이름
- 공유: 이미지 저장, 인쇄, 실시간 링크
- 복구: 자동저장, 이름 붙여 저장, JSON 백업

## 새 종목으로 확장할 때 질문

개발 시작 전에 아래만 정하면 빠르다.

1. 한 경기당 몇 명이 필요한가?
2. 팀/개인전이 모두 필요한가?
3. 선수별 최소/목표 경기 수는 무엇인가?
4. 실력 균형은 개인 레벨 합으로 보면 되는가?
5. 성별/연령/급수 같은 조합 제한이 있는가?
6. 경기 중 선수 추가/제외를 허용할 것인가?
7. 관전자에게 실시간으로 보여줄 핵심 정보는 무엇인가?
8. 배포는 GitHub Pages 같은 정적 웹으로 충분한가, 서버가 필요한가?
