# 콕매치 개발 요약

이 문서는 콕매치와 유사한 동호회/경기 운영 프로그램을 빠르게 만들기 위한 재사용 요약입니다.

## 현재 프로젝트

- 이름: 콕매치
- 용도: 배드민턴 동호회 민턴LIVE 운영, 팀전LIVE 운영, 출석/상태 입력, 자동 대진, 코트 진행, 승패 입력, 실시간 공유
- 배포: GitHub Pages
- 운영 URL: https://kimminhyun22.github.io/badminton/
- Firebase 프로젝트: `kokmatch-23b31`
- 주요 파일:
  - `index.html`: 민턴LIVE 관리자 화면, 명부·링크·코트·대진·마무리 운영
  - `team.html`: 팀전LIVE 관리자 화면, 명부·링크·청홍팀·대진표·승패 운영
  - `checkin.html`: 민턴LIVE 회원 화면, 내 상태와 다음 대진 확인
  - `rsvp.html`: 팀전LIVE 회원 화면, 출석·늦음·게스트·실시간 현황 확인
  - `view.html`: 카톡 공유용 실시간 중계 화면
  - `manifest.json`, `sw.js`: PWA 설치와 캐시
  - `icon-192.png`, `icon-512.png`, `apple-touch-icon.png`: 앱 아이콘

## 버전 규칙

코드를 수정하면 `scripts/bump-version.js`로 앱 전체 버전을 함께 올린다.

- HTML의 `APP VERSION`, `<meta name="app-version">`, 화면 표시 버전
- `js/daily.js`, `js/team.js`, `js/live-view.js`의 `APP_VERSION`
- `sw.js` 캐시 버전
- `README.md` 현재 버전

예: `1.10.83` -> `1.10.84`

## 핵심 운영 철학

1. 현장 운영자는 비개발자라고 가정한다.
2. 버튼 한 번으로 바로 쓸 수 있어야 한다.
3. 좋은 대진을 만들기 위해 사용자가 여러 번 생성 버튼을 누르게 하지 않는다.
4. 품질점수는 단순 점수보다 실전 해석이 중요하다.
5. 불가피한 반복과 실제로 피할 수 있는 문제를 구분한다.
6. 스마트폰 가독성을 최우선으로 둔다.
7. 경기 중에는 복잡한 기능보다 실수 방지가 더 중요하다.
8. 민턴LIVE와 팀전LIVE를 분리하되, 둘 다 “현장 흐름을 끊지 않는 자동화”를 기준으로 둔다.
   - 민턴LIVE: 명부 → 링크 공유 → 코트 수 → 진행 중 경기 등록 → 자동대진 → 마무리 → 자율게임
   - 팀전LIVE: 명부 → 링크 공유 → 30분 현장 확인 → 참가자 불러오기 → 청/홍 배정 → 대진 생성 → 승패 입력
   - 관리자는 전체 메뉴를 훑지 않고 현재 해야 할 버튼과 핵심 현황만 본다.
   - 회원 화면은 마이페이지를 우선 보여주고, 내 상태와 내 다음 행동을 가장 크게 보여준다.

## 대진 생성 주요 기준

- 실력 균형: 양 팀 실력 합 차이를 최소화한다.
- 출전 공정성: 인당 목표 게임 수 달성을 가장 중요하게 본다.
- 파트너 다양성: 같은 파트너 반복을 줄인다. 단, 사용자가 지정한 P 파트너는 예외로 본다.
- 재대결 다양성: 같은 4명이 다시 만나는 경기와 완전히 같은 팀 대결을 줄인다.
- 휴식 간격: 직전 라운드 연속 출전을 줄인다.
- 일정 효율성: 불필요한 보완게임을 줄인다.
- 설정 준수: 남복/여복/혼복/보정 경기 조건이 맞는지 확인한다.

## 민턴LIVE 특수 규칙

- `시작`은 지금 경기할 준비가 됐다는 뜻이다.
- `휴식`, `종료`, `경기중`은 실제 투입 후보에서 제외한다.
- 경기 중인 선수도 예상 대진 후보에는 일부 포함해 대기 불안을 줄인다.
- 예상 대진은 확정이 아니며, 경기 종료·휴식·종료·게임신청에 따라 바뀔 수 있다.
- 진행 중 경기 선수는 현실적으로 휴식/종료를 누르기 어렵기 때문에 경기 종료 후 반영한다.
- 다음 대진 1순위가 쉬면 후순위 또는 예상 대진에서 자동 보충한다.
- `이번만 뒤로`는 한 칸만 밀리는 양보이며, 순서를 크게 잃지 않는다.
- `마무리`는 새 자동대진 생성을 멈추고 남은 대진만 진행한 뒤 자율게임으로 넘긴다.
- 회원 게임신청은 2명 또는 4명이 모두 준비됐을 때 자동 반영한다.

## 팀전LIVE 특수 규칙

- 링크는 출석 확인, 대진표 확인, 실시간 현황 확인을 하나로 이어준다.
- 실전에서는 링크 공유 후 바로 대진을 만들지 않고, 난타 중 최종 참가자와 게스트를 확인한 뒤 만든다.
- 늦음 선수는 대진에는 포함하되 라운드 시작 전 대체 가능성만 확인한다.
- 팀전LIVE 운영 화면은 중복 요약보다 현재 라운드, 입력 필요 코트, 승패 확인 필요만 보여준다.
- 진행 중 대진과 다음 대진은 모두 승패 입력이 가능해야 한다.
- 전체 초기화는 링크 응답, 게스트, 참가자, 팀배정, 대진표, 승패를 지우되 클럽 명부는 남긴다.

## 실시간 중계 화면 설계

카톡 링크로 들어오는 선수가 주로 궁금해하는 것:

- 내 이름이 맞게 선택됐는지
- 내 상태가 반영됐는지
- 내가 다음에 무엇을 해야 하는지
- 지금 진행 중인 경기와 다음/예상 대진이 무엇인지

따라서 회원 화면은 아래 순서가 좋다.

1. 내 카드
2. 내 다음 대진 또는 내 다음 행동
3. 진행 중 경기
4. 다음 대진
5. 예상 대진
6. 필요한 경우 전체 현황

스마트폰 가독성을 위해 카드형 UI, 큰 선수 이름, 명확한 팀 색상, 접이식 상세 정보가 적합하다.

## 유사 프로그램을 빠르게 만들 때 재사용할 구조

### 1. 데이터 모델

필수 엔티티:

- 참가자: 이름, 성별, 레벨, 상태, 게스트 여부, 묶음 요청, 최근 대기시간
- 경기: 라운드, 코트, 종목, 팀1 선수 2명, 팀2 선수 2명, 승패
- 설정: 코트 수, 자동배정 여부, 마감 모드, 마감 예약
- 기록: 출전 수, 승패, 파트너 카운트, 상대 카운트

### 2. 생성 알고리즘

추천 흐름:

1. 참가자와 설정 검증
2. 여러 후보 대진 생성
3. 각 후보를 품질점수로 평가
4. 가장 좋은 후보 자동 선택
5. 미달 선수 보완
6. 품질 대시보드와 실전 특이사항 표시

### 3. 품질점수 설계

점수는 운영자에게 쉬워야 한다.

- 숫자만 보여주지 말고 이유를 함께 보여준다.
- “불가피한 문제”와 “재배정하면 개선 가능한 문제”를 나눈다.
- 실전 특이사항에는 사람 이름과 경기 수를 직접 보여준다.

### 4. UX 원칙

- PC: 입력과 결과를 한 화면에서 빠르게 확인
- 모바일: 하단 탭, 큰 버튼, 큰 선수 이름
- 공유: 이미지 저장, 인쇄, 실시간 링크
- 복구: 자동저장, 이름 붙여 저장, JSON 백업

## 새 종목으로 확장할 때 질문

개발 시작 전에 아래만 정하면 빠르다.

1. 한 경기당 몇 명이 필요한가?
2. 팀/개인전이 모두 필요한가?
3. 선수별 최소/목표 경기 수는 무엇인가?
4. 실력 균형은 개인 레벨 합으로 보면 되는가?
5. 성별/연령/급수 같은 조합 제한이 있는가?
6. 경기 중 선수 추가/제외를 허용할 것인가?
7. 관전자에게 실시간으로 보여줄 핵심 정보는 무엇인가?
8. 배포는 GitHub Pages 같은 정적 웹으로 충분한가, 서버가 필요한가?
